1. Ansible — Install & Enable Nginx

Correct Playbook
---
- name: Install and start Nginx on Pathnex hosts
  hosts: all
  become: yes

  tasks:
    - name: Install nginx
      yum:
        name: nginx
        state: present

    - name: Enable and start nginx service
      service:
        name: nginx
        state: started
        enabled: yes
🔍 Why we use this

Ansible is used for configuration management.

Step-by-step:

hosts: all → runs on all servers
become: yes → runs with root privileges
yum → installs nginx package
service → starts nginx immediately and ensures it starts on reboot

👉 Purpose: Automate server setup without manual SSH work.

☁️ 2. Terraform — EC2 with Tags
✔ Correct Terraform Code
provider "aws" {
  region = "us-east-1"
}

resource "aws_instance" "PathnexEC2" {
  ami           = "ami-0abcd1234abcd1234"
  instance_type = "r5.2xlarge"

  tags = {

🔍 Why we use this

Terraform is used for Infrastructure as Code (IaC).

Step-by-step:

provider → tells AWS region
resource aws_instance → creates EC2 VM
ami → base OS image
instance_type → hardware size
tags → metadata for tracking resources

👉 Purpose:

Create cloud infrastructure automatically
Track everything as code (version controlled)
☸️ 3. Kubernetes — Deployment (2 replicas)
✔ Correct YAML
apiVersion: apps/v1
kind: Deployment
metadata:
  name: pathnex-deployment

spec:
  replicas: 2
  selector:
    matchLabels:
      app: pathnex-app

  template:
    metadata:
      labels:
        app: pathnex-app

    spec:
      containers:
        - name: app
          image: nginx
          ports:
            - containerPort: 80
🔍 Why we use this

Kubernetes is used for container orchestration.

Step-by-step:

replicas: 2 → runs 2 identical pods (high availability)
selector → links deployment to pods
template → defines pod blueprint
nginx image → container runs web server
containerPort: 80 → exposes HTTP port inside container

👉 Purpose:

Auto-healing (if pod dies, new one is created)
Load balancing between replicas
Scalable deployments

⚙️ 5. Jenkins Pipeline — Maven Build
✔ Correct Jenkinsfile
pipeline {
    agent any

    tools {
        maven 'Maven-3.8.1'
        jdk 'JDK-17'
    }

    stages {

        stage('Checkout') {
            steps {
                git url: 'https://github.com/Pathnex/sample-java-app.git'
            }
        }

        stage('Compile') {
            steps {
                sh 'mvn clean compile'
            }
        }

        stage('Test') {
            steps {
                sh 'mvn test'
            }
        }

        stage('Package') {
            steps {
                sh 'mvn package'
            }
        }
    }
}
🔍 Why we use this

Jenkins is used for CI/CD automation.

Step-by-step:

Checkout → pulls code from GitHub
Compile → converts Java code into bytecode
Test → runs unit tests (ensures code works)
Package → builds .jar or .war file

👉 Purpose:

Automate software build process
Ensure every code change is tested
Produce deployable artifact
🧪 6. GitLab CI — Maven Pipeline
✔ Correct .gitlab-ci.yml
stages:
  - build
  - test
  - package

maven-build:
  stage: build
  image: maven:3.8.1-jdk-17
  script:
    - git clone https://github.com/Pathnex/sample-java-app.git
    - cd sample-java-app
    - mvn clean compile

maven-test:
  stage: test
  image: maven:3.8.1-jdk-17
  script:
    - cd sample-java-app
    - mvn test

maven-package:
  stage: package
  image: maven:3.8.1-jdk-17
  script:
    - cd sample-java-app
    - mvn package
🔍 Why we use this

GitLab CI is also for CI/CD automation but inside GitLab.

Step-by-step:

stages → defines pipeline flow
build → compiles code
test → validates logic
package → creates deployable artifact
image: maven → ensures Maven environment is available

Purpose:

Fully automated pipeline inside GitLab
No need separate Jenkins server
Consistent builds using containers
